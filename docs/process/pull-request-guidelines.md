# Pull Request Guidelines

A good pull request is clear, verifiable, and easy to review.

## Required PR contents

- Summary of what changed
- Problem being solved and why now
- Implementation approach and key decisions
- Testing evidence (what was run and results)
- Documentation updates
- Risks, assumptions, and follow-up actions

## Scope and size

- Keep PRs small enough for effective review.
- Avoid bundling unrelated changes.
- Split large initiatives into sequenced PRs when possible.

## Testing expectations

- Include tests for behavior changes.
- Include regression tests for bug fixes where practical.
- Report any known untested areas explicitly.

## Documentation expectations

- Update docs when behavior, structure, or process changes.
- Link to relevant docs in PR description.

## Reviewer experience

- Use clear commit messages.
- Structure PR description for quick understanding.
- Call out high-risk areas and assumptions early.

## Merge readiness checklist

- CI checks passing
- Required approvals obtained
- Open review comments resolved
- Scope remains aligned with PR description
