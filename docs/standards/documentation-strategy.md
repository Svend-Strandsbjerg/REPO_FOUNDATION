# Documentation Strategy

This strategy defines documentation expectations for repositories using this foundation.

## Documentation principles

- Documentation is part of the deliverable, not an afterthought.
- Keep docs accurate, concise, and discoverable.
- Prefer practical guidance over exhaustive prose.

## Required baseline docs

### README

- Explain repository purpose and scope.
- Provide quick-start orientation for contributors.
- Outline high-level workflow and quality expectations.

### Architecture documentation

- Capture operating model and structural decisions.
- Record governance expectations (branching, PRs, CI, review).
- Update when process or architecture decisions change.

### Process/standards documentation

- Define repeatable workflows and contribution practices.
- Establish coding, testing, and documentation standards.
- Keep expectations consistent across contributors.

## Update triggers

Update documentation when any of the following changes:

- Behavior or interfaces
- Repository structure
- Development/review process
- Testing strategy or quality gates

## AI-assisted contribution expectations

- AI-generated changes must include documentation updates where relevant.
- Contributors must verify generated docs for correctness and consistency.
- PRs should explicitly state what documentation was updated.
